# 3.2 Разбор вызовов

Ниже описано, что происходит при запуске каждой команды (`mkdir`, `ls`, `rmdir`, `mkfile`, `read`, `rm`, `symlink`, `readlink`, `linkdata`, `rmlink`, `hard`, `rmhard`, `stat`, `chmod`) в программе `2.c`.

Разбор идет по слоям:

1. Пользовательский код и libc (glbic/musl wrappers).
2. Переход в ядро (системный вызов).
3. VFS (Virtual File System).
4. Конкретная файловая система (ext4/xfs/btrfs и т.д.).
5. Блочный слой и устройство хранения (SSD/HDD/NVMe).

## 1. Общий путь выполнения процесса

Когда вы запускаете, например, `./mkdir dir1`, ОС делает следующее:

1. Shell вызывает `fork()`.
2. Дочерний процесс вызывает `execve("./mkdir", argv, envp)`.
3. Ядро загружает ELF-бинарник, отображает сегменты в память (`mmap`/demand paging), настраивает стек процесса, передает управление в `_start`.
4. В рантайме C инициализируются libc, аргументы `argc/argv`, потом вызывается `main`.
5. В `main` берется `argv[0]`, через `strrchr` определяется имя команды, после чего `switch(parse_cmd(cmd))` выбирает ветку.

Важно: `parse_cmd`, `strcmp`, `strrchr` работают в user-space и **не** делают системных вызовов сами по себе.

## 2. Как в целом проходит любой файловый системный вызов

Для большинства команд путь внутри ядра схож:

1. **Копирование аргументов из user-space** (`copy_from_user`).
2. **Path walk**: разбор пути по компонентам (`/a/b/c`) через dentry cache (dcache).
3. **Проверки доступа** (UID/GID, mode bits, ACL, capabilities, LSM: SELinux/AppArmor).
4. **Операция VFS** (создать inode, удалить dentry, прочитать атрибуты и т.п.).
5. Делегирование в методы конкретной ФС (`inode_operations`, `file_operations`, `super_operations`).
6. При необходимости работа с page cache, журналом (например, jbd2 у ext4), аллокатором блоков.
7. Возврат кода ошибки или успеха в user-space (`errno` при ошибке).

## 3. По командам

## `mkdir <path>`

Код: `mkdir(argv[1], 0755)`.

### Системный вызов

- libc вызывает `mkdirat(AT_FDCWD, path, 0755)` (на современных Linux обычно именно `mkdirat`).

### Что делает ядро

1. Разбирает родительский каталог (`path_lookupat`).
2. Проверяет права на родительский каталог: нужен `write + execute` на директорию.
3. Проверяет, что имя не занято.
4. Создает новый inode типа directory.
5. Создает dentry-имя в родительской директории.
6. Инициализирует `.` и `..` (логически/физически в зависимости от ФС).

### Что делает драйвер ФС

- Резервирует метаданные (inode table/extent metadata).
- Обновляет каталог-родитель (добавляет запись имени).
- Обновляет счетчики ссылок (`st_nlink`) у новой директории и родителя.
- Для журналируемых ФС пишет транзакцию в журнал (например, ext4+jbd2), затем коммитит.

## `ls <dir>`

Код:

- `opendir(argv[1])`
- цикл `readdir(d)`
- `closedir(d)`

### Системные вызовы

- `opendir` -> `openat(..., O_RDONLY|O_DIRECTORY)`.
- `readdir` обычно читает порциями через `getdents64`.
- `closedir` -> `close`.

### Что делает ядро

1. Открывает директорию, создает файловый дескриптор и `struct file`.
2. При `getdents64` вызывает итератор каталога ФС.
3. Копирует записи каталога в буфер user-space (`linux_dirent64`).

### Что делает драйвер ФС

- Читает блоки каталога (часто через page cache).
- Декодирует внутренний формат каталога (например, ext4 dir entries + htree индексация).
- Возвращает последовательность имен и inode номеров.

## `rmdir <path>`

Код: `rmdir(argv[1])`.

### Системный вызов

- `rmdir` -> `unlinkat(AT_FDCWD, path, AT_REMOVEDIR)`.

### Что делает ядро

1. Проверяет, что цель существует и это директория.
2. Проверяет, что директория пуста (обычно только `.` и `..`).
3. Проверяет права на родителя.
4. Удаляет запись имени из родителя.
5. Уменьшает link count директории и родителя.

### Что делает драйвер ФС

- Изменяет структуру каталога-родителя.
- Помечает inode директории свободным, когда счетчик ссылок и открытий достигает нуля.
- Фиксирует метаданные через журнал/транзакцию.

## `mkfile <path>`

Код: `open(argv[1], O_CREAT | O_WRONLY, 0644)`, затем `close(fd)`.

### Системный вызов

- `open` обычно превращается в `openat(AT_FDCWD, path, O_CREAT|O_WRONLY, 0644)`.

### Что делает ядро

1. Path walk до родителя и имени.
2. Если файла нет: создает inode регулярного файла.
3. Если файл есть: просто открывает (потому что нет `O_EXCL`, нет `O_TRUNC`).
4. Выделяет fd в таблице дескрипторов процесса.

### Что делает драйвер ФС

- При создании: выделяет inode и запись в каталоге.
- Данные файла не аллоцируются, потому что запись в файл не выполняется.
- Метаданные (ctime/mtime/atime в зависимости от mount-опций) обновляются.

## `read <path>`

Код:

- `open(path, O_RDONLY)`
- цикл `read(fd, buf, 4096)`
- `write(1, buf, n)`
- `close(fd)`

### Системные вызовы

- `openat`, многократные `read`, многократные `write` в stdout, `close`.

### Что делает ядро

1. `openat`: обычный путь открытия.
2. `read`:
   - проверка fd и прав на чтение;
   - попытка отдать данные из page cache;
   - при cache miss инициируется I/O к устройству;
   - копирование данных в user буфер.
3. `write(1, ...)`: запись в stdout (терминал/pipe/файл, в зависимости от перенаправления).

### Что делает драйвер ФС

- Организует чтение блоков файла по отображению offset -> physical blocks/extents.
- Заполняет page cache страницами файла.
- Для повторных чтений часто обслуживает без обращения к диску.

## `rm <path>`, `rmlink <path>`, `rmhard <path>`

В коде все три команды приводятся к `unlink(argv[1])`.

### Системный вызов

- `unlink` -> `unlinkat(AT_FDCWD, path, 0)`.

### Что делает ядро

1. Находит запись имени в родительском каталоге.
2. Удаляет именно **имя** (directory entry), а не «объект как таковой» напрямую.
3. Уменьшает `st_nlink` inode.
4. Если ссылок стало 0 и файл никем не открыт, освобождает inode/блоки.

### Что делает драйвер ФС

- Изменяет каталог-родитель (удаление dir entry).
- Обновляет inode и bitmap/extents.
- При последней ссылке освобождает блоки данных и метаданные.

Примечание: для симлинка `unlink` удаляет сам симлинк-файл, не целевой файл.

## `symlink <target> <linkpath>`

Код: `symlink(argv[1], argv[2])`.

### Системный вызов

- `symlink` -> `symlinkat(target, AT_FDCWD, linkpath)`.

### Что делает ядро

1. Разбирает путь `linkpath` и проверяет родительский каталог.
2. Создает inode типа symbolic link.
3. Сохраняет строку `target` как содержимое симлинка.

### Что делает драйвер ФС

- Для коротких target-строк может хранить путь прямо в inode (fast symlink).
- Для длинных строк выделяет отдельные блоки данных.
- Добавляет запись `linkpath` в каталог.

## `readlink <linkpath>`

Код: `readlink(argv[1], buf, sizeof(buf)-1)` и печать.

### Системный вызов

- `readlink` -> `readlinkat(AT_FDCWD, linkpath, buf, bufsz)`.

### Что делает ядро

1. Находит inode симлинка без финального follow ссылки.
2. Читает «сырое» содержимое симлинка (путь target).
3. Копирует байты в user буфер.

Особенность: `readlink` **не** добавляет `\0`, это делает ваш код (`buf[n]=0`).

### Что делает драйвер ФС

- Извлекает строку target из inode/блоков данных симлинка.

## `linkdata <linkpath>`

Код:

1. `readlink(linkpath, buf, ...)` -> получаем строку target.
2. `open(buf, O_RDONLY)` -> открываем путь target как обычный путь.
3. `read`/`write` как в команде `read`.

### Низкоуровнево

- Первая часть повторяет `readlink`.
- Вторая часть повторяет `open+read`.
- Здесь есть семантическая деталь: открывается путь, который вернул симлинк, **относительно текущей рабочей директории процесса**, а не относительно директории, где лежит симлинк. Это может отличаться от обычного разрешения симлинков ядром при `open(linkpath, ...)`.

### Риски

- TOCTOU: между `readlink` и `open` target может быть подменен.
- Нет проверки длины/нормализации пути сверх размера буфера.

## `hard <oldpath> <newpath>`

Код: `link(argv[1], argv[2])`.

### Системный вызов

- `link` -> `linkat(AT_FDCWD, oldpath, AT_FDCWD, newpath, 0)`.

### Что делает ядро

1. Находит inode `oldpath`.
2. Проверяет ограничения: обычно нельзя hardlink на директории (кроме спецслучаев), нужно быть в той же ФС.
3. Добавляет новое имя `newpath`, указывающее на тот же inode.
4. Увеличивает `st_nlink`.

### Что делает драйвер ФС

- Добавляет запись в каталог `newpath`.
- Обновляет inode (link count, ctime).

## `stat <path>`

Код: `stat(argv[1], &st)` и печать `st_mode & 0777`, `st_nlink`.

### Системный вызов

- `stat` обычно идет через `newfstatat(AT_FDCWD, path, &st, 0)`.

### Что делает ядро

1. Path walk и разрешение (для `stat` симлинк обычно разыменовывается).
2. Чтение inode-атрибутов.
3. Преобразование внутренних полей ядра в структуру `struct stat` ABI.

### Что делает драйвер ФС

- Возвращает метаданные inode: права, uid/gid, размер, времена, число ссылок.
- Часто берется из inode cache; при промахе читается с диска.

## `chmod <path> <octal_mode>`

Код: `chmod(argv[1], strtol(argv[2], NULL, 8))`.

### Системный вызов

- `chmod` -> `fchmodat(AT_FDCWD, path, mode, 0)`.

### Что делает ядро

1. Path walk и проверка прав (владелец файла или CAP_FOWNER и др.).
2. Обновляет mode bits inode.
3. Обновляет ctime.

### Что делает драйвер ФС

- Записывает измененные метаданные inode.
- Фиксирует изменения в журнале/на носителе согласно политике writeback.

## `bad cmd`

Если `argv[0]` не совпал ни с одной строкой в таблице:

- системных вызовов ФС нет;
- идет `fprintf(stderr, ...)` (это приводит к `write(2, ...)` в stderr);
- процесс завершает `main` с кодом 1.

## 4. Что происходит в `close(fd)`

Практически в каждой команде есть `close`.

1. Ядро уменьшает счетчик ссылок на `struct file`.
2. Если это последняя ссылка, вызывается `file_operations->release`.
3. Для файлов на запись могут быть триггеры flush/sync в зависимости от состояния страниц и mount-опций.

`close` не гарантирует физическую запись на диск немедленно; для гарантии используется `fsync`/`fdatasync`.

## 5. Роль page cache и writeback

Для чтения:

- Сначала проверка page cache.
- Если страницы есть в памяти, чтение обходится без реального I/O.

Для метаданных и записи:

- Изменения часто сначала попадают в память (dirty pages/dirty metadata).
- Фоновый writeback поток ядра (`flush-*`, `jbd2/*` для ext4 журнала) позже пишет на носитель.

## 6. Журналирование и crash-consistency

На журналируемых ФС (например, ext4 по умолчанию):

1. Операции с метаданными оформляются как транзакции журнала.
2. Сначала в журнал, затем commit, потом checkpoint в основную структуру ФС.
3. После сбоя при монтировании выполняется replay журнала.

Это уменьшает риск повреждения структуры ФС, но не всегда гарантирует сохранность пользовательских данных без `fsync`.

## 7. Где именно «драйвер файловой системы»

Технически в Linux обычно разделяют:

1. VFS слой (общий для всех ФС интерфейс).
2. Код конкретной ФС (ext4/xfs/btrfs/f2fs) — это и есть то, что чаще имеют в виду как «драйвер ФС».
3. Блочный слой (`bio`, request queue, I/O scheduler).
4. Драйвер устройства (NVMe/AHCI/SCSI) и сам накопитель.

Ваши команды доходят до уровня 2 почти всегда, а до уровня 4 — когда нужны реальные чтение/запись, которых нет в кэше.

## 8. Нюансы именно этой реализации `2.c`

1. Нет проверок `argc`, поэтому при нехватке аргументов возможен segfault до системного вызова.
2. Почти не проверяются коды ошибок syscalls, из-за чего не видна причина сбоя (`errno`).
3. `mkfile` без `O_TRUNC` не очищает существующий файл.
4. `linkdata` логически отличается от `cat symlink` по разрешению относительных путей.
5. `read_file` пишет в fd `1`; если stdout перенаправлен, запись пойдет в файл/pipe.

## 9. Как посмотреть это на практике

Для верификации цепочки можно использовать:

1. `strace -f ./mkdir testdir` — список syscalls user-space процесса.
2. `perf trace` — агрегированный runtime trace.
3. `bpftrace`/eBPF (kprobes/tracepoints) — что происходит в VFS/FS функциях.
4. `blktrace`/`iostat` — выход на блочный слой и устройство.

Так можно увидеть переходы от вашего C-кода до операций на уровне ядра и диска.
