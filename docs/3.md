# Устройство файловой системы

## 1) Общая картина: VFS и конкретные файловые системы

В Linux есть слой **VFS (Virtual File System)** — единый интерфейс для приложений.

- Приложение вызывает `open/read/write/stat`.
- VFS делает путь-независимую часть работы (разрешение пути, кэш dentry/inode, проверки прав).
- Затем операция передаётся в драйвер конкретной ФС (например, ext4).

Идея: одинаковый API снаружи, разные реализации хранения внутри.

---

## 2) Directory (`dir`) — каталог как файл специального типа

Каталог в Unix/Linux — это специальный файл, который хранит отображение:

- **имя файла** → **номер inode**.

Каталог **не хранит сами данные файлов**. Он хранит только имена и ссылки на inode.

### Что важно
- В каждом каталоге есть `.` (ссылка на себя) и `..` (ссылка на родительский каталог).
- У каталога есть собственный inode, права доступа и временные метки.
- Операции `readdir` читают записи каталога (directory entries).

---

## 3) `direntry` / `dentry` — запись каталога и объект кэша имён

Термин может использоваться в двух смыслах:

1. **On-disk directory entry** (на диске):
   - структура в каталоге с именем и номером inode.

2. **VFS dentry** (в памяти ядра):
   - объект кэша «компонент пути ↔ inode».

### Зачем нужен dentry-кэш
Разрешение пути `/a/b/c.txt` требует прохода по каждому компоненту. Dentry-кэш ускоряет это:

- повторные обращения к тем же путям не читают каталог с диска заново;
- поддерживаются и «отрицательные» dentry (имя не найдено), чтобы ускорять повторные `ENOENT`.

---

## 4) `inode` — метаданные объекта

`inode` (index node) — структура, описывающая объект ФС:

- тип (обычный файл, каталог, симлинк, устройство, сокет, FIFO),
- права (`rwx` + спецбиты),
- UID/GID, = > владелец, группа,
- размер,
- метки времени (`atime`, `mtime`, `ctime`, иногда `btime`),
- счётчик жёстких ссылок,
- указатели/экстенты на блоки данных.

### Что inode **не** хранит
- имя файла.

Имя хранится в записи каталога (direntry). Поэтому один inode может иметь несколько имён (жёсткие ссылки).

---

## 5) Hard link и symlink

## Hard link
- Ещё одно имя для **того же inode**.
- Увеличивает `nlink`.
- Обычно нельзя делать на каталоги (для целостности дерева).
- Если создавать hard link на каталог, то в дереве появятся циклы или несколько путей к одному каталогу, что нарушит древовидную структуру и сломает алгоритмы обхода дерева (например, `find` или удаление).
- Работает только в пределах одной ФС.

## Symlink
- Отдельный inode типа «символическая ссылка».
- Хранит путь-цель как строку.
- Может ссылаться через разные ФС.
- Может быть «битой» (цель отсутствует).

---
## 6) Superblock, block groups, bitmaps, journal

### Superblock
Хранит глобальные параметры ФС:
- размер блока (обычно 4 КБ),
- общее число inode и блоков,
- состояние ФС (чистая/грязная после перепада питания),
- UUID и метка тома,
- признаки возможностей (features) и требования (требования для ядра).

Ядро держит in-memory представление superblock для каждого смонтированного экземпляра ФС. Это позволяет быстро узнать параметры без чтения с диска.

### Группы блоков (block groups, типично для ext*)
Для улучшения локальности данных и снижения фрагментации ФС делится на группы:

- **Блокбитмап**: битовая карта, где каждый бит обозначает занят ли блок данных (1 бит = 1 блок).
- **Inode-битмап**: битовая карта для inode (1 бит = 1 доступный номер inode).
- **Таблица inode**: физический массив структур inode для этой группы (здесь хранятся сами inode на диске).
- **Область данных**: блоки, где хранятся содержимое файлов и метаданные.

**Зачем это нужно:** Ядро может быстро выделить место для нового файла в той же группе, где лежит его каталог. Это уменьшает фрагментацию и ускоряет доступ (локальность).

### Журнал (journaling)
В ext4, xfs и других журналируемых ФС метаданные записываются в журнал **перед** применением к диску:

- **Защита от сбоев**: если произойдёт сбой питания посреди записи, журнал содержит полную транзакцию, которую можно откатить или завершить.
- **Быстрое восстановление**: fsck не сканирует весь диск; ядро просто проигрывает (replay) незавершённые операции из журнала.
- **Типы журналирования**:
   - **Metadata-only**: только метаданные в журнал (быстро).
   - **Data+metadata**: и данные, и метаданные (надёжнее, но медленнее).
   - **Ordered** (по умолчанию в ext4): данные пишутся перед метаданными, минимизируя риск несогласованностей.

## 7) File object и таблицы дескрипторов

При `open()` создаются/используются уровни:

1. **fd table процесса**: число `fd` → указатель на file object.
2. **open file description (`struct file`)**: текущая позиция, флаги (`O_APPEND`...), ссылка на dentry/inode.
3. **inode**: метаданные + операции ФС.

Следствия:

- `dup()` даёт новый `fd` на тот же `struct file` (общая позиция).
- После `fork()` ребёнок наследует `fd`, указывающие на те же open file descriptions.

Когда вы в коде на C пишете int fd = open(...), число fd — это просто индекс в массиве.

У каждого процесса есть структура task_struct.

В ней есть таблица открытых файлов (files_struct).

Ваш fd — это номер ячейки в этой таблице.

В этой ячейке лежит указатель на конкретную struct file.
---

## 8) Path resolution (namei)

Разрешение пути в ядре (упрощённо):

1. старт из `/` или текущего каталога,
2. разбор компонента,
3. поиск/кэширование dentry,
4. проверка прав доступа,
5. переход к следующему компоненту,
6. обработка симлинков (с ограничением глубины).

Именно здесь активно работают dentry/inode-кэши.

---

## 9) Кэширование в Linux I/O

## Page cache
Данные файлов кэшируются в памяти страниц:

- `read` часто обслуживается из RAM,
- `write` обычно помечает страницы «грязными» и сбрасывает позже.

## Inode cache / dentry cache
Кэш метаданных и имён снижает число обращений к диску.

## Buffering и sync
- `fsync(fd)` — принудительно сбросить данные/метаданные файла.
- `sync`/`syncfs` — более широкий сброс.

---

## 10) Монтирование и namespace

`mount` подключает ФС к дереву каталогов в точку монтирования.

- После монтирования путь работает в едином дереве.
- Разные ФС выглядят единообразно через VFS.
- В Linux есть mount namespaces (изоляция дерева монтирования, важно для контейнеров).

---

## 11) Минимальная модель «как всё связано»

Путь `./dir/file.txt`:

- `dir` (каталог) содержит запись `file.txt -> inode N`.
- inode `N` хранит метаданные и ссылки на блоки данных.
- Данные читаются/пишутся через page cache и драйвер ФС.
- VFS использует dentry/inode cache, чтобы ускорить повторный доступ.

---

## 12) Коротко по терминам

- **dir (directory)**: файл-каталог, таблица имён → inode.
- **direntry (on-disk)**: запись каталога (имя + inode number).
- **dentry (VFS)**: in-memory объект для кэша имён и путей.
- **inode**: метаданные объекта + адреса данных.
- **superblock**: глобальные параметры конкретной ФС.
- **file (`struct file`)**: состояние открытого файла (позиция, флаги).
- **fd**: номер дескриптора в процессе.

---

## 13) Практическая связь с системными вызовами

- `open(path)` → path resolution (dentry/inode) + создание `struct file` + возврат `fd`.
- `stat(path)` → чтение inode-метаданных.
- `readdir(dirfd)` → чтение direntries каталога.
- `read/write` → работа через page cache и операции ФС.
- `unlink` → удаление direntry; inode удаляется, когда `nlink == 0` и нет открытых ссылок.
- `rename` → атомарная операция на уровне каталожных записей (в рамках одной ФС).

## 14) Что возвращает opendir, что такое DIR*
- `opendir(path)` возвращает указатель на `DIR` — структуру, которая используется для чтения содержимого каталога через `readdir`.
- `DIR*` — это указатель на структуру, которая содержит информацию о текущем положении в каталоге и может использоваться для итерации по записям каталога.

## 15) dirent*
- `readdir(DIR* dirp)` возвращает указатель на `struct dirent`, который представляет собой запись каталога, содержащую имя файла и номер inode.
- `struct dirent` обычно содержит поля `d_name` (имя файла) и `d_ino` (номер inode), а также другие поля, зависящие от конкретной реализации.

## 16) pread/pwrite
- `pread(fd, buf, count, offset)` — читает `count` байт из файла, открытого через `fd`, начиная с позиции `offset`, не изменяя текущую позицию в файле.
- `pwrite(fd, buf, count, offset)` — записывает `count` байт в файл, открытый через `fd`, начиная с позиции `offset`, не изменяя текущую позицию в файле.
- Эти функции полезны для многопоточных операций чтения/записи, когда нужно работать с разными частями файла без изменения общей позиции.
- Когда вы открываете файл через open(), операционная система создает в ядре таблицу, где хранится текущая позиция (offset) для этого дескриптора.
- Обычный read/write: Сначала считывает данные, а затем автоматически сдвигает указатель вперед на количество прочитанных байт. Многопоточные операции с одним `fd` могут привести к гонкам, поэтому `pread/pwrite` позволяют избежать этого.

## 17) Где хранятся inode
### На диске
- **ext4/ext3/ext2**: inode хранятся в *таблице inode* (inode table) в каждой группе блоков (block group).
   - Местоположение вычисляется по номеру inode: `group = (ino - 1) / inodes_per_group`, затем смещение внутри таблицы.
   Ограничение: максимальное число inode задаётся при форматировании ФС.
   Как посмотреть: `df -i` покажет использование inode.

- **xfs/btrfs**: используют собственные структуры (может быть встроено в B-tree или отдельные структуры).
   Динамическое выделение inode позволяет создавать больше файлов, чем фиксированное число при форматировании.

### В памяти
- Ядро кэширует inode в *inode cache* (VFS-слой).
- При обращении к файлу inode считывается с диска один раз и держится в памяти.

### Как прочитать inode для файла

**Способ 1: системный вызов `stat()`**
```c
#include <sys/stat.h>
struct stat sb;
stat("/path/to/file", &sb);
// sb.st_mode    — права и тип файла
// sb.st_ino     — номер inode
// sb.st_nlink   — счётчик жёстких ссылок
// sb.st_size    — размер
// sb.st_uid/gid — владелец/группа
// sb.st_mtime   — время модификации
```

**Способ 2: команда `stat`**
```bash
stat /path/to/file
# выведет: Access: (0644/-rw-r--r--), Uid: 1000, Gid: 1000, Size: 1234, Inode: 12345, ...
```

**Способ 3: `ls -i` — вывести номер inode**
```bash
ls -i /path/to/file
# 12345 /path/to/file
```

**Способ 4: прямое чтение на диске (требует root)**
```bash
debugfs -R "stat <inode_number>" /dev/sdXY
# покажет все метаданные inode из таблицы inode на диске
```

```
debugfs:  stat 1.c
Inode: 408372   Type: regular    Mode:  0644   Flags: 0x80000
Generation: 3521193239    Version: 0x00000000:00000006
User:  1000   Group:  1000   Project:     0   Size: 2969
File ACL: 0
Links: 1   Blockcount: 8
Fragment:  Address: 0    Number: 0    Size: 0
 ctime: 0x69b23b7d:c1d9ba04 -- Thu Mar 12 11:05:17 2026
 atime: 0x69b23763:af0f30d8 -- Thu Mar 12 10:47:47 2026
 mtime: 0x69b23b7d:c1d9ba04 -- Thu Mar 12 11:05:17 2026
crtime: 0x69b23763:af0f30d8 -- Thu Mar 12 10:47:47 2026
Size of extra inode fields: 32
Inode checksum: 0x3ba6f37f

(0):120003769
```

- `Inode: 408372` — номер inode.
- `Type: regular` — тип файла (обычный).
- `Mode: 0644` — права доступа.
- `User: 1000` — UID владельца.
- `Group: 1000` — GID группы.
- `Size: 2969` — размер файла в байтах.
- `Links: 1` — число жёстких ссылок (nlink).
- `Blockcount: 8` — количество блоков, выделенных для данных.
- `ctime/atime/mtime` — метки времени изменения метаданных, доступа и модификации.
- `crtime` - создание файла (creation time).
-  `Size of extra inode fields: 32`
   Эта строка указывает на размер «дополнительного» пространства внутри иноды.
   В ext4 стандартная инода обычно занимает 256 байт.
   Базовая часть (совместимая с ext2/3) занимает 128 байт.
   Остальное место используется для расширенных атрибутов (Extended Attributes) и тех самых новых полей, как crtime.
   Значение 32 говорит о том, что сразу после основной структуры выделено 32 байта под расширенные поля метаданных
- `Inode checksum: 0x3ba6f37f` — контрольная сумма иноды для проверки целостности данных.
   Это механизм защиты от повреждения данных (bit rot) или ошибок записи на диск.
   Как это работает: Файловая система вычисляет хеш (обычно CRC32C) от всех данных в этой иноде и записывает его сюда. При следующем чтении система снова считает хеш и сравнивает его с этим значением.
   Если они не совпадают, ядро понимает, что метаданные файла повреждены, и может вовремя остановить работу или запустить восстановление, не давая ошибке распространиться.

- `EXTENTS` — список экстентов (для ext4), показывающий, какие блоки на диске выделены для данных файла.


Flags: 0x80000: Этот флаг означает EXTENTS_FL. Файл использует экстенты (непрерывные цепочки блоков) вместо старой системы косвенной адресации блоков. Это значительно ускоряет работу с большими файлами.

В ext2 и ext3 каждый блок файла (обычно 4 КБ) записывался отдельно. Если у вас файл размером 100 МБ, системе нужно было хранить список из 25 600 номеров блоков.

- Минус: Огромные накладные расходы на метаданные.
- Минус: Сильная фрагментация.
- Минус: Чтобы прочитать файл, нужно сначала прочитать кучу «указателей» на блоки.

2. Как работают экстенты
- Экстент — это один дескриптор, который говорит: «Файл начинается с блока X и занимает N идущих подряд блоков».
- Стандартный экстент в ext4 описывается 12 байтами и включает:
- Логический номер блока (где в файле начинается этот кусок).
- Длину (сколько блоков идет подряд, максимум 32 768 блоков или 128 МБ на один экстент).
- Физический адрес (где именно на диске лежат эти данные).

3. Дерево экстентов (Extent Tree)
- Инода (inode) имеет ограниченный размер. В ней помещается заголовок и 4 экстента.
- Если файл маленький (до 4 кусков), все данные о его местоположении лежат прямо в иноде. Это очень быстро!
- Если файл большой или сильно фрагментирован, экстенты выстраиваются в дерево (B-tree). Инода указывает на «узлы-индексы», которые ведут к «листьям» с реальными адресами.

4. Главные преимущества
- Компактность: Вместо списка из тысяч блоков — всего одна или несколько записей.
- Производительность: Файловая система старается выделять блоки группами. Меньше движений головок HDD и меньше работы для контроллера SSD.
- Эффективность при удалении: Удалить файл на 10 ГБ гораздо быстрее, если нужно пометить свободными 80 экстентов, а не 2,5 миллиона отдельных блоков.
